# Cyber Court 产品优化计划

## 1. 当前结论

当前项目适合演示概念，不适合直接作为正式产品上线。

原因不是“功能少”这么简单，而是它仍然停留在单页 Demo 阶段：

- UI 有视觉主题，但信息层级单薄，缺少产品首页、帮助、状态反馈、空态、失败态和可信度表达。
- 前端把输入、配置、请求、结果全部堆在一个页面中，可维护性和扩展性较差。
- 后端只有一个直连模型的 API 路由，没有鉴权、限流、日志、重试、结构化校验、成本控制和观测能力。
- 安全模型不成立。当前把第三方 API Key 存在 `localStorage`，这对面向真实用户的产品不可接受。
- 文案与代码存在编码损坏，说明工程链路和文件编码管理不稳定。
- 缺少上线前必需的 SEO、埋点、异常监控、测试、合规文案和运营闭环。

如果目标是“真正上线并可持续迭代”，建议按下面的分阶段计划执行，而不是继续在现有页面上零散打补丁。

## 2. 现状问题清单

### 2.1 UI / 产品层

- 当前只有一个主流程页，缺少产品落地页、价值说明和用户引导。
- 输入区是大文本框直贴，缺少示例模板、输入规范、平台说明、隐私提示。
- 分析态只有仪表盘动画，等待体验单一，没有阶段反馈和预期时间提示。
- 结果页信息结构单一，只展示比分和文案，缺少摘要、证据点、逻辑谬误标签、可分享卡片体系。
- 文案风格有趣，但“毒舌”过强，缺少产品级内容边界与风险控制。
- 没有移动端细节校准、可访问性校验和品牌一致性。

### 2.2 前端工程层

- `app/page.tsx` 承担过多状态和展示逻辑，后续加功能会迅速失控。
- API 配置、核心业务状态、结果展示没有形成独立 domain。
- 缺少表单校验、请求状态管理、统一错误处理和数据 schema。
- 当前大量 inline style，主题复用和设计系统能力弱。
- 组件边界偏粗，后续做多场景复用成本高。
- 缺少测试和 Storybook/组件文档类支撑。
- 已出现中文乱码，需统一 UTF-8 编码并清理受损文案。

### 2.3 后端 / 平台层

- 只有单一路由 `app/api/verdict/route.ts`，职责过重。
- 直接透传用户 API Key 与 Base URL，不具备正式产品的安全边界。
- 缺少输入清洗、输出 schema 校验、超时控制、重试策略和 provider 适配层。
- 没有用户系统、配额、任务记录、结果持久化、分享链接、风控和审计。
- 没有监控、日志、报警、性能指标和成本分析。

## 3. 产品化目标

优化目标不是“把页面做得更炫”，而是把项目升级成一个可上线、可运营、可扩展的 AI Web 产品。

核心目标：

- 用户能快速理解产品价值，并安全地完成一次分析。
- 系统能稳定输出结构化结果，而不是纯粹依赖模型自由发挥。
- 前端具备可维护的模块结构，后续可以继续加模板、历史记录、分享、登录等能力。
- 后端具备基础安全、限流、监控和容错能力。
- 上线后能够观察真实使用情况，并持续迭代。

## 4. 分阶段优化路线

---

## Phase 0：基础修复与产品基线

这是必须先做的阶段，不做这一步，后面的迭代会建立在不稳定基础上。

### UI / 内容

- 全量修复乱码文案，统一中文语气和品牌术语。
- 重新定义品牌表达：保留“赛博法庭”的趣味性，但降低攻击性，避免过度冒犯。
- 补齐空态、失败态、加载态、无结果态文案。

### 前端

- 统一文件编码为 UTF-8。
- 拆分 `page.tsx` 的业务逻辑，至少拆成：
  - 页面容器层
  - 输入表单模块
  - 分析状态模块
  - 结果展示模块
  - 本地配置模块
- 引入 schema 校验层，约束请求参数和返回数据。
- 整理公共类型定义，避免结果对象在页面和 API 之间松散流动。

### 后端

- 给 `verdict` 接口补齐：
  - 请求参数校验
  - 模型返回结构校验
  - 超时和错误分类
  - provider 异常兜底
- 收敛 prompt 和结果 schema，降低随机输出带来的不稳定性。

### 交付标准

- 全站无乱码。
- 核心流程能稳定跑通。
- 错误信息对用户可理解，对开发者可定位。

---

## Phase 1：UI 体系升级

目标是把当前“好玩的单页”提升为“有产品感的体验”。

### 页面结构重构

- 新增 Landing 页面，明确产品价值、使用方式、示例和隐私承诺。
- 将“分析器”做成独立主工作区，而不是首页唯一内容。
- 增加结果页内的信息分区：
  - 判决摘要
  - 双方画像
  - 关键证据
  - 逻辑谬误标签
  - AI 判词
  - 分享卡片

### 设计系统

- 抽取颜色、间距、圆角、阴影、文字层级为 design tokens。
- 降低 inline style 比例，建立可复用样式体系。
- 统一按钮、输入框、卡片、标签、提示条、弹窗规范。
- 优化移动端排版与触控尺寸。

### 交互体验

- 输入区增加粘贴示例、格式提示和字数建议。
- 分析中展示步骤式进度反馈，而不是纯随机仪表动画。
- 结果区支持分段 reveal，而不是一次性砸出所有内容。
- 海报生成改成更完整的分享体验，支持多模板。

### 可访问性

- 提高颜色对比度，保证文本可读性。
- 完善按钮语义、焦点态和键盘可达性。
- 给图表提供文本替代说明。

### 交付标准

- 首屏更像产品而不是代码作品集。
- 移动端和桌面端都能稳定使用。
- 主要组件具备一致性和复用能力。

---

## Phase 2：前端架构重构

目标是让项目能承接后续增长，而不是继续把需求堆进一个页面。

### 目录与模块

- 重新组织前端目录，例如：
  - `app/(marketing)`
  - `app/analyze`
  - `app/history`
  - `components/ui`
  - `features/verdict`
  - `features/api-settings`
  - `lib/`
- 把业务模块按 feature 拆分，而不是继续按“散组件”堆叠。

### 状态管理

- 区分本地 UI 状态、分析任务状态和用户配置状态。
- 引入更清晰的请求状态模型：idle / validating / submitting / success / error。
- 对用户输入做预处理，例如：
  - 自动识别对话角色
  - 清理无效空行
  - 限制极端长文本

### 数据与边界

- 在客户端和服务端共享 schema。
- 统一 fetch 封装，标准化错误对象。
- 明确 Server Components 和 Client Components 的边界，避免页面客户端负担过重。

### 质量保障

- 为关键组件补测试。
- 为核心流程补集成测试。
- 给关键交互和结果模块建立视觉回归基线。

### 交付标准

- 主流程代码可读、可测、可扩展。
- 新增功能不需要继续修改超大页面文件。

---

## Phase 3：后端服务化与安全改造

这是决定能否正式上线的关键阶段。

### 安全模型重做

- 正式产品不再依赖用户自己填写第三方 API Key。
- 改为服务端托管模型调用，用户只与本产品后端交互。
- 增加基本的滥用防护：
  - IP / 账号级限流
  - 请求频率控制
  - 输入长度限制
  - 风险内容过滤

### 服务层设计

- 抽出 AI provider service，不让 route 直接拼装所有逻辑。
- 拆分以下职责：
  - 输入解析
  - Prompt 构建
  - 模型调用
  - 输出校验
  - 结果规范化
- 为不同模型供应商预留适配层。

### 数据持久化

- 建立基础数据库表：
  - users
  - analysis_jobs
  - analysis_results
  - share_cards
  - usage_logs
- 支持结果回看、分享页和基础运营分析。

### 稳定性与观测

- 接入错误监控和请求日志。
- 记录模型耗时、失败率、token 成本、返回质量。
- 为任务设置超时、降级和重试策略。

### 交付标准

- 核心接口具备生产级错误处理和可观测性。
- 不再把安全边界建立在浏览器本地存储上。

## Phase 4：产品功能补齐

当基础稳定后，再做真正能提升留存和传播的功能。

### 用户侧功能

- 历史记录与再次查看
- 分享页 / 公共结果页
- 多种分析模式：
  - 情侣吵架
  - 朋友争执
  - 职场沟通
  - 群聊判案
- 输入模板与示例库
- 多语言支持

### 产品增长能力

- 首次使用引导
- 分享激励
- SEO 页面
- FAQ、使用说明、免责声明

### 交付标准

- 产品不只是“跑通一次”，而是具备复用和传播能力。

---

## Phase 5：上线准备

### 工程与运维

- 环境变量管理
- 发布流程
- 监控与报警
- 备份策略
- 性能优化

### 合规与内容治理

- 隐私政策
- 用户协议
- AI 输出免责声明
- 不当内容处理策略

### 数据与运营

- 埋点设计
- 转化漏斗
- 留存与复访指标
- 热门模板与失败场景分析

## 5. 推荐执行顺序

建议严格按下面顺序推进：

1. Phase 0：先把乱码、结构混乱、接口脆弱这些基础问题清掉。
2. Phase 1：升级 UI 和交互，让它从 Demo 变成产品体验。
3. Phase 2：重构前端架构，为持续迭代打底。
4. Phase 3：重做后端安全和服务层，具备真实上线条件。
5. Phase 4：补历史记录、分享、模板等增长功能。
6. Phase 5：做监控、合规、埋点和正式发布。

## 6. 下一阶段的具体实施建议

下一轮优化建议不要直接做“加功能”，而是优先完成一个最小但正确的第一批改造：

### 第一批必须完成

- 修复全站乱码与文案体系
- 重构首页信息架构
- 拆分 `page.tsx`
- 给 `/api/verdict` 增加 schema 校验和错误治理
- 去掉“用户本地保存第三方 API Key”作为默认产品路径

### 第二批紧接着做

- 设计系统和基础组件统一
- 分析结果结构升级
- Landing 页和帮助说明页
- 基础日志、埋点、监控

## 7. 我们后续的执行方式

后续优化建议按“文档驱动 + 分阶段交付”进行。

每一轮只做一个明确阶段，避免同时改 UI、架构和后端导致范围失控。

推荐节奏：

1. 先完成 Phase 0。
2. 我输出并落地 Phase 1 的具体页面与组件改造。
3. 再进入 Phase 2 和 Phase 3 的结构性改造。

## 8. 当前项目中的高优先级风险

- 中文乱码已经污染代码和元数据，必须立即修复。
- API Key 本地存储只适合个人自用，不适合公开产品。
- 模型输出解析过于脆弱，缺少严格校验。
- 页面单文件承载过多业务，继续迭代会加速失控。
- README 仍是默认模板，说明项目对外说明和工程基线尚未建立。

## 9. 结论

如果只是做作品展示，这个项目已经有记忆点。

如果要作为正式产品上线，它现在确实偏简陋，而且问题不只在 UI，而是在整条产品链路都还没形成。正确做法不是局部美化，而是按阶段把 UI、前端架构、后端服务、安全和上线能力一起补齐。
